# セキュリティモデル

## 脅威分析

### 脅威1: 不正アクセス
- **リスク**: 第三者がシステムにアクセス
- **対策**: Firebase Auth + Googleログイン + メール許可リスト
- **検証**: 許可リスト外のメールでログイン試行 → 拒否されること

### 脅威2: APIキー漏洩
- **リスク**: APIキーがブラウザやGitに露出
- **対策**:
  - APIキーはサーバーサイド（Cloud API, Local Worker）のみで保持
  - ブラウザ（PWA）にはAPIキーを渡さない
  - .envファイルは.gitignoreに追加
  - .env.exampleのみコミット
  - Secret Manager使用（本番時）
- **検証**: ソースコード全体をスキャンしてAPIキーパターンが存在しないこと

### 脅威3: プロンプトインジェクション
- **リスク**: 悪意ある入力でAIの動作を操作
- **対策**:
  - ユーザー入力をサニタイズ
  - システムプロンプトとユーザー入力を構造的に分離
  - .env, 秘密情報をAIに渡さない
  - AIの出力も検証
- **検証**: インジェクション文字列テスト

### 脅威4: 破壊的操作の誤実行
- **リスク**: 本番DB削除、force pushなど
- **対策**:
  - 必須承認操作リスト
  - 承認なしでは実行不可
  - すべての承認を監査ログに記録
- **検証**: 承認なしで危険操作を試行 → ブロックされること

### 脅威5: コマンド実行の悪用
- **リスク**: AIが任意のシェルコマンドを実行
- **対策**:
  - コマンド許可リスト (ホワイトリスト)
  - コマンド拒否リスト (ブラックリスト)
  - 実行タイムアウト
  - ワークスペース外へのアクセス禁止
- **検証**: 拒否リストのコマンド実行試行 → ブロックされること

### 脅威6: ファイルシステムへの不正アクセス
- **リスク**: ワークスペース外のファイルにアクセス
- **対策**:
  - パス検証（ワークスペース内に限定）
  - .env, 認証情報ファイルへのアクセスブロック
  - シンボリックリンク経由のエスケープ防止
- **検証**: ワークスペース外パスでの操作試行 → 拒否されること

### 脅威7: 無制限のAI利用
- **リスク**: 意図しない高額課金
- **対策**:
  - タスクごとのコスト上限
  - 月間コスト上限
  - トークン使用量の記録
  - 上限超過時は人間にエスカレーション
- **検証**: 上限を低く設定して超過テスト

## 必須承認操作

以下は明示的な人間の承認なしに実行禁止:

1. 本番環境へのデプロイ
2. 本番DBの変更
3. DBマイグレーション
4. DBデータ削除
5. 大量ファイル削除（10ファイル以上）
6. Git force push
7. mainブランチへの直接push
8. PRのマージ
9. 秘密情報の変更
10. 課金設定の変更
11. 外部サービスの新規作成
12. 新しい有料リソースの作成
13. 月間予算を超えるAI呼び出し
14. ロールバック
15. ユーザーデータへ影響する処理

## 監査ログ

すべての承認操作を記録:
- **誰が**: ユーザーIDとメール
- **いつ**: タイムスタンプ (ISO 8601)
- **何を**: 操作の種類と対象
- **結果**: 承認/却下/エラー
- **理由**: ユーザーのコメント（あれば）
