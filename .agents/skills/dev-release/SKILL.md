# dev-release — リリース管理

> **スラッシュコマンド:** `/dev-release`
>
> デプロイ前チェック・バックアップ・リリースノートを作成し、安全なリリースを実現します。

---

## 概要

`/dev-release` は、コードを本番環境にデプロイするための最終チェックと準備を行います。

---

## 絶対ルール

> [!CAUTION]
> **本番環境へのデプロイは、必ず人間の承認を得てから実行すること。**
> 自動デプロイは禁止。承認なしのデプロイは即座に中止する。

---

## 前提条件

- `/dev-build` が完了していること
- `/dev-review` で承認済みであること
- PRが作成済みであること

---

## 実行手順

### Step 1: デプロイ前チェックリスト

以下の項目をすべて確認:

#### 1.1 コード品質チェック

- [ ] すべてのテストがパスしている
- [ ] レビューが承認済み
- [ ] Lintエラーがゼロ
- [ ] 型エラーがゼロ
- [ ] セキュリティ findings がすべて解決済み

#### 1.2 Git状態チェック

```bash
# PR の状態確認
gh pr status

# main との差分確認
git diff main...devcore/{task-id}-{slug} --stat

# コンフリクトの確認
git merge-tree $(git merge-base main HEAD) main HEAD
```

- [ ] PRがオープンかつレビュー承認済み
- [ ] main とのコンフリクトなし
- [ ] 不要なファイルが含まれていない

#### 1.3 環境変数チェック

- [ ] 新しい環境変数が追加される場合、本番環境に設定済みか確認
- [ ] `.env.example` が更新されているか
- [ ] 機密情報がコードに含まれていないか

#### 1.4 データベースチェック

- [ ] マイグレーションファイルが存在する場合、内容を確認
- [ ] マイグレーションがロールバック可能か
- [ ] データの整合性に影響がないか
- [ ] マイグレーション実行順序が正しいか

### Step 2: バックアップ確認

```json
{
  "backup_checklist": {
    "database": {
      "required": true,
      "method": "pg_dump / mongodump / Cloud SQL export",
      "status": "pending_human_confirmation",
      "note": "デプロイ前にDBバックアップを取得してください"
    },
    "files": {
      "required": false,
      "note": "ユーザーアップロードファイルがある場合はバックアップ"
    },
    "config": {
      "required": true,
      "method": "環境変数・設定ファイルのスナップショット",
      "status": "pending"
    }
  }
}
```

### Step 3: マイグレーション計画

マイグレーションが必要な場合:

```json
{
  "migration_plan": {
    "has_migrations": true,
    "migrations": [
      {
        "name": "20250115_add_avatar_url",
        "type": "schema_change",
        "description": "usersテーブルにavatar_urlカラムを追加",
        "reversible": true,
        "rollback_command": "npx prisma migrate rollback",
        "data_loss_risk": false,
        "estimated_duration": "< 1秒",
        "downtime_required": false
      }
    ],
    "execution_order": [
      "1. バックアップ取得",
      "2. マイグレーション実行",
      "3. アプリケーションデプロイ",
      "4. 動作確認"
    ]
  }
}
```

### Step 4: リリースノート作成

```markdown
# リリースノート — TASK-002

## バージョン: v1.2.0
## リリース日: 2025-01-15

### 新機能
- ユーザープロフィール編集機能を追加
  - 表示名の変更
  - メールアドレスの変更（確認メール付き）
  - プロフィール画像のアップロード

### 変更点
- プロフィールページにeditボタンを追加
- ユーザーAPIに PUT /api/user/profile を追加

### バグ修正
- なし

### 破壊的変更
- なし

### マイグレーション
- usersテーブルにavatar_urlカラムを追加（自動マイグレーション）

### 環境変数
- `CLOUD_STORAGE_BUCKET` — プロフィール画像の保存先バケット（追加必須）

### デプロイ手順
1. 環境変数 `CLOUD_STORAGE_BUCKET` を設定
2. DBマイグレーションの実行: `npx prisma migrate deploy`
3. アプリケーションのデプロイ
4. 動作確認

### ロールバック手順
1. アプリケーションを前バージョンにロールバック
2. DBマイグレーションのロールバック: `npx prisma migrate rollback`
3. 環境変数はそのまま（害なし）
```

### Step 6: デプロイ実行（承認後のみ）

人間の承認を受けた後のみ実行:

1. **PRのマージ**: main ブランチにマージ
2. **デプロイトリガー**: CI/CDパイプラインの実行（または手動デプロイ）
3. **マイグレーション実行**: DBマイグレーションの実行
4. **動作確認**: デプロイ後の基本動作確認
5. **モニタリング**: エラー率・レスポンスタイムの監視

## デプロイ後の処理

1. **プロジェクトメモリの更新**:
   - `current-task.json` のステータスを `deployed` に
   - `task-history.json` にタスクを移動
2. **ブランチの整理**:
   - マージ済みブランチの削除（ローカル・リモート）
3. **次のタスクの準備**:
   - `current-task.json` をクリア

---

## 注意事項

- 金曜日の遅い時間のデプロイは推奨しない（問題発生時の対応困難）
- ステージング環境でのテストを推奨（環境が存在する場合）
- デプロイ後最低30分はモニタリングを行う
- ロールバック手順は事前にテストしておくことを推奨
