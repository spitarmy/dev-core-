# dev-build — コード実装

> **スラッシュコマンド:** `/dev-build`
>
> 仕様書に基づいてコードを実装し、テストを実行します。

---

## 概要

`/dev-build` は、`/dev-plan` で作成された仕様書に基づいて以下を実行します:

1. Gitブランチの作成
2. コードの実装
3. テストの実行
4. 失敗時の自動修正（最大3回）
5. プロジェクトメモリの更新

---

## 前提条件

- `/dev-init` が完了していること（`.devcore/` が存在する）
- `/dev-plan` で仕様書が作成済みであること
- 該当タスクの仕様がユーザーに承認済みであること

---

## 実行手順

### Step 1: コンテキストのロード

`project-memory` スキルを使用して以下をロード:

1. `current-task.json` — タスク情報
2. `specs/{task-id}.json` — 仕様書
3. `coding-rules.json` — コーディング規約
4. `tech-stack.json` — 技術スタック
5. `danger-zones.json` — 危険ゾーン情報

### Step 2: モデルの選定

`model-routing` スキルを使用して実装モデルを選定:

```json
{
  "task_type": "implementation",
  "complexity": "仕様書から判定",
  "context": {
    "is_google_cloud": "tech-stackから判定",
    "file_count": "影響ファイル数",
    "estimated_tokens": "推定トークン数"
  }
}
```

選定されたモデルを `current-task.json` に記録。

### Step 3: Gitブランチの作成

`git-workflow` スキルを使用してブランチを作成:

```bash
# ブランチ名の規則: devcore/{task-id}-{slug}
git checkout -b devcore/TASK-002-profile-edit

# slug の生成規則:
# - タスクタイトルから英語のキーワードを抽出
# - 小文字、ハイフン区切り
# - 最大30文字
```

### Step 4: コードの実装

以下の原則に従って実装:

#### 実装原則

1. **最小限の変更（Minimal Change Principle）**
   - 仕様に記載された機能のみ実装する
   - 「ついでに」のリファクタリングは行わない
   - 既存コードの不要な変更は避ける

2. **既存ルール遵守**
   - `coding-rules.json` に記載された規約を守る
   - 既存コードのスタイル・パターンに合わせる
   - lint / formatter の設定に従う

3. **危険ゾーンの注意**
   - 危険ゾーンに該当するファイルを変更する場合は警告を表示
   - `requires_human_approval: true` のゾーンは人間の承認を得てから変更

4. **テストコード必須**
   - すべての新規機能に対してテストを作成
   - テストは最低限、単体テストを含む
   - テストファイルの命名は既存の規約に従う

#### 実装の進行

各実装ステップ完了時にチェックポイントを記録:

```json
{
  "checkpoint_id": "CP-001",
  "timestamp": "ISO8601",
  "phase": "implementation",
  "step": 1,
  "description": "DBマイグレーション作成完了",
  "files_modified": ["prisma/migrations/20250115_add_avatar_url/migration.sql"],
  "status": "completed"
}
```

### Step 5: テストの実行

`testing` スキルを使用してテストを実行:

1. **Lint チェック** — コード品質の確認
2. **型チェック** — TypeScript等の型エラー確認
3. **単体テスト** — 新規・変更コードのテスト
4. **結合テスト** — API連携・データフローのテスト
5. **既存テスト** — 回帰テスト（既存テストが壊れていないか）

### Step 6: 自動修正ループ（最大3回）

テスト失敗時の自動修正フロー:

```
テスト実行
  ↓
失敗? → いいえ → Step 7 へ
  ↓ はい
修正試行 1
  ↓
テスト再実行
  ↓
失敗? → いいえ → Step 7 へ
  ↓ はい
修正試行 2
  ↓
テスト再実行
  ↓
失敗? → いいえ → Step 7 へ
  ↓ はい
修正試行 3（最終）
  ↓
テスト再実行
  ↓
失敗? → いいえ → Step 7 へ
  ↓ はい
エスカレーション → ユーザーに報告
```

#### 自動修正の原則

- **エラーメッセージを分析**して原因を特定
- **最小限の変更**で修正（大規模な書き換えは避ける）
- **各修正の内容を記録**する
- 3回失敗後は以下を報告:
  - 元のエラー内容
  - 各修正試行の内容と結果
  - 推測される根本原因
  - 手動修正の提案

### Step 7: 実装完了処理

1. **すべてのテストがパスしていることを確認**
2. **変更ファイルの一覧を作成**
3. **Git コミット**（意味のあるコミットメッセージ）:
   ```
   feat(TASK-002): ユーザープロフィール編集機能を追加

   - プロフィール更新APIの作成
   - 編集フォームコンポーネントの作成
   - 画像アップロード処理の実装
   - 単体テスト・結合テストの追加
   ```
4. **プロジェクトメモリの更新**:
   - `current-task.json` のステータスを `implementation_complete` に
   - チェックポイントの最終更新
5. **ユーザーへの報告**:
   - 実装サマリー
   - 変更ファイルリスト
   - テスト結果
   - 次のステップ（`/dev-review` の提案）

---

## 仕様矛盾の検知

実装中に仕様の矛盾を発見した場合:

1. **即座に実装を中断**
2. **矛盾の内容を記録**
3. **ユーザーに報告**:
   ```
   ⚠️ 仕様の矛盾を検知しました:
   
   仕様AC-002: 「メールアドレス変更時、確認メールが送信される」
   問題: メール送信サービスが未設定です。
   
   選択肢:
   1. メール送信サービスのセットアップを追加する
   2. メールアドレス変更機能をスコープ外にする
   3. 仮実装（ログ出力のみ）で進める
   ```

---

## 注意事項

- `main` / `master` ブランチへの直接コミットは**禁止**
- 実装中は定期的にチェックポイントを記録する
- 大規模な変更は複数のコミットに分割する
- センシティブ情報（API Key等）のハードコードは禁止
